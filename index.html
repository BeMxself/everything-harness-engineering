What Is Harness Engineering

这里的 harness engineering 不是单指“某个 agent 会不会写代码”，而是指： 如何为 agent 搭建一整套工程化外壳，让它能在真实工作流中稳定运行。

它关注什么

  • 任务如何进入系统，如何被拆解、委派、恢复和完成
  • agent 如何受规则、工具、状态、审批、验证与记忆层约束
  • 如何把“聪明模型”变成“能连续产出结果的工程系统”

它不等于什么

  • 不等于单次 prompt engineering
  • 不等于某个 IDE 插件本身
  • 不等于只会写 demo 的 autonomous agent

为什么现在重要

  • 模型能力已经足够强，真正的瓶颈开始转向系统化约束与执行环境
  • 团队需要的是可重复、可验证、可恢复、可协作的 agent 工程链路
  • “agent 能做什么”正在快速让位于“你如何 harness 它”

Harness vs Harness-Like Suites

边界不是绝对的，但从工程角度看，可以先按下面三类来理解生态。

1. Native Harness

直接围绕 agent 的执行闭环搭建：任务输入、工具调用、状态管理、验证、重试、委派、权限与环境控制都在同一工程外壳内。

典型问题：如何把一个 coding agent 变成真实工程产线的一部分？

2. Harness-Like Agent Suites

不一定把自己叫 harness，但实际已经在做类似事情：规则层、协作层、上下文管理、子代理、工作流约束、恢复机制。

典型问题：如何让 agent 在复杂 repo 与复杂组织规则里持续工作？

3. Workflow / Orchestration Suites

更强调流程编排、图结构、工作流节点、deterministic + agent 混合编排，有时不专注 coding，但解决的是相邻问题。

典型问题：如何让多个 agent、工具、人类审批与系统事件一起组成稳定流程？

Current Landscape

当前生态不是一条直线，而是多个方向同时演化。下面这个列表不是“绝对权威排名”，而是一个面向工程观察的分类视图。

Native Coding Harnesses / Shells

  • Claude Code
  • Codex
  • OpenCode
  • Goose
  • Aider

这些工具更接近“直接给 agent 一个真实 coding 工作面”，强调终端、repo、工具和执行闭环。

Harness-Like Engineering Suites

  • oh-my-opencode
  • OpenHands
  • Langroid
  • Harness Kit

这些项目不一定都把自己定义为 harness，但在规则层、协作层、上下文管理与恢复策略上，已经呈现出明显的 harness engineering 特征。

Workflow / Orchestration Suites

  • LangGraph
  • Microsoft Agent Framework
  • CrewAI
  • GitHub Agentic Workflows

这类系统更偏流程图、工作流与多 agent 编排，不一定专注 coding harness，但解决的是相邻问题：如何让 agent 系统稳定进入工程流程。
