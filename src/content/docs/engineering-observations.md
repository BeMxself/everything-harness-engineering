这一页收录的不是官方定义，而是本仓库从 [关键文章](../key-articles/) 与 [社区讨论](../community-discussions/) 里反复看到的稳定工程信号。它们更接近 `engineering interpretation`，不是 `canonical fact`。

## 这些观察是怎么形成的

- 它们不是来自单一厂商，也不是来自单一社区帖子，而是来自多份材料里反复出现的相似问题。
- 它们的价值不在“证明某个做法唯一正确”，而在帮助读者辨认哪些问题一旦进入长时 agent 工作流，就几乎一定会冒出来。
- 因此，这一页最适合配合 [概念入口](../concept-entry/) 一起看：概念页告诉你系统由哪些层构成，这一页告诉你真实实践里哪一层最容易先变重。

## 控制面往往比代码面更重

一个反复出现的信号是：在长任务里，真正占据系统复杂度的往往不是代码改了多少，而是如何规划、交接、记录、审查和恢复。很多时间花在让 agent 持续处于“知道自己在干什么、下一步该干什么”的状态上。

### 为什么这是稳定信号

Anthropic 的长任务文章把 progress artifact、clean handoff 和 context reset 提到核心位置；OpenAI 的 harness 文章则把 legibility、guardrails 和 merge discipline 视为工程对象。这些都说明：一旦任务跨过单轮会话，控制面就不再是附属品。

### 这会改变什么

- 站点在比较项目时，不能只比较“会不会改代码”，还要比较 `谁在控制下一步`
- 读者在评估一个 harness 时，应该重点看记录、审批、恢复和可读性，而不是只看模型能力
- 这也是为什么本仓库把 `host`、`memory`、`approval` 和 `handoff` 当成研究对象，而不是产品外围细节

### 边界

短任务、一次性脚本或高约束自动化流里，控制面未必会显著重于代码面。这个观察最适用于持续多轮推进、需要验证和交接的工程任务。

## 长任务的关键是 clean handoff

多份材料都在强调：如果任务要跨会话、跨上下文窗口甚至跨 agent 继续，`clean handoff` 就会从“好习惯”变成基础设施要求。没有交接结构，系统很快会退化回普通问答。

### 为什么这是稳定信号

Anthropic 把 clean handoff 说成长期 agent 成败关键；社区实践里则不断回到“计划文件”“进度摘要”“续跑文档”这些具体工件。这两边实际上在说同一件事：恢复能力不是回放历史，而是压缩出可继续工作的状态。

### 这会改变什么

- 好的 harness 不只是能暂停和继续，而是能在继续时减少重新理解成本
- `memory` 的价值不在存更多，而在把下轮真正需要的控制信息留下来
- 这也是 [宿主与 Harness 结构图](../topics/) 这类专题值得存在的原因之一：它们能帮助读者看到恢复路径是回到哪里，而不只是回到“再试一次”

### 边界

并不是所有系统都需要重 handoff 设计。高度同步、短链路、单控制器任务里，简单回溯历史也可能够用。问题一旦变成长任务或多人协同，handoff 的权重就会迅速上升。

## Single controller + document API

社区实践里反复出现一个工程直觉：高阶决策层不应该直接淹没在原始代码噪声里，更有效的做法是保留一个干净的 controller，再通过文档、摘要、计划、图表和报告去压缩信息。

### 为什么这是稳定信号

OpenAI 的 harness 表达里一直强调 `humans steer, agents execute`；很多社区工作流也把文档当作不同 agent 或不同层级模型之间的接口。换句话说，文档不只是给人看的，它还是控制层之间交换状态的 API。

### 这会改变什么

- 研究 harness 时，文档系统应该被看成运行机制的一部分，而不是附属产物
- 这也解释了为什么技能系统、spec-driven system、review discipline 常常会与 harness 套件黏在一起出现
- 对本站来说，这条观察会继续影响后续 topic 设计：专题不只是展示结果，也应该展示“中间工件如何承担控制作用”

### 边界

这不是在说“所有任务都应该文档化到极致”。问题的关键不是文档数量，而是有没有一种可复用、可压缩、可交接的信息表面。

## 模型供应现实会反过来塑造 harness

很多 harness 实践并不是稳定 doctrine，而是在模型能力、供应稳定性、成本、上下文窗口和工具可用性变化下不断漂移。模型升级会让一部分脚手架失去必要性，也会暴露新的瓶颈。

### 为什么这是稳定信号

官方文章和社区经验都在不断调整关注点：一段时间大家强调 prompt discipline，之后重心移到 tools，再后来变成 memory、handoff、approval、protocol。问题不是“以前谁错了”，而是瓶颈位置真的在移动。

### 这会改变什么

- 本站的分类和图谱必须允许漂移，不能把今天的实现形态写成永久定论
- 研究页面里需要明确标注哪些是 `fact`，哪些是当前阶段的 `interpretation`
- 这也说明专题页面比纯列表更重要，因为它们能把“为什么现在这样分”解释清楚

### 边界

供应现实会影响 harness，但不会自动决定一切。好的设计仍然要能区分：什么是临时适配，什么是长期有效的工程对象。
